“记忆”和“缓存”在多智能体项目里经常被混用。教学时如果不先把这两个词拆开，学生看到 Redis、PostgreSQL 和 LangGraph state 会很容易误判它们是同一种东西。

本项目把三类状态并列展示，目的是让你能直接回答一个问题：某条信息到底该保留多久，又是为了什么保留。

## Example

假设用户多次请求“请用中文、按教学结构解释 Redis”。我们观察同一偏好在不同层里的存在方式。

### Step 1: 先看 LangGraph state

LangGraph state 只在一次工作流里存在。它记录当前 run 的计划、研究摘要、草稿和审查结果。

如果这次运行结束，state 本身就应该消失。它的用途是支撑当前流程，不负责跨会话记忆。

这也是为什么 `GraphState` 更像“运行现场”，而不是知识库。

### Step 2: 再看 PostgreSQL 长期记忆

长期记忆要解决的是“下次还能记住”。例如用户偏好默认用中文回答，这个信息就应该保存到 PostgreSQL。

`app/memory/long_term.py` 负责把这些条目保存成结构化记录。下次新的任务启动时，可以先读取这些记忆，再把摘要拼进 prompt。

这里的摩擦点在于：如果你把长期记忆也只放在内存里，一重启服务就全没了。PostgreSQL 的意义就在这里。

### Step 3: 最后看 Redis 缓存

缓存不是为了“记住用户”，而是为了“少做重复工作”。例如某次研究结果、某个热点查询、某段昂贵的外部调用结果，都适合放进 Redis。

一旦缓存 miss，系统仍然可以继续工作，只是更慢。长期记忆 miss 则意味着系统忘了过去的上下文，这两者影响完全不同。
